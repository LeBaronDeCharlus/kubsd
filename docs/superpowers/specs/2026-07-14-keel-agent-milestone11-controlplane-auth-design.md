# Milestone 11: Control-Plane Authentication (Sub-Project 3, Third Milestone)

Status: Approved
Date: 2026-07-14

## Context

Milestones 7-10 built the multi-node control plane (`keel-controlplane`'s node
registry, request forwarding, scheduling, and resource-aware placement) under
a trust model each of those specs stated explicitly and identically:
"same-network trust is assumed... hardening this is explicitly deferred."
Concretely, today, anyone who can reach `keel-controlplane`'s TCP port or a
`keel-agentd`'s opt-in `--advertise-addr` TCP listener can register a fake
node, spoof another node's heartbeat, or apply/delete a jail on any node in
the cluster, with no credential of any kind. This is the one gap identified,
while assessing what "production ready" would require next, as an actual
liability rather than a missing nice-to-have: every other undesigned item on
the roadmap (cluster networking, storage beyond one host, bhyve) is a scope
expansion, not something making the existing system unsafe to run.

This milestone closes that gap for *authorization* (who is allowed to call
these endpoints). It deliberately does not close it for *confidentiality*
(encrypting the wire against an on-path eavesdropper on the same LAN
segment) â€” that is real transport security, warrants pulling in an actual
TLS implementation rather than hand-rolling one, and is scoped as its own
separately-designed future milestone. Splitting the two mirrors this
project's usual milestone size (each of Milestones 7-10 shipped one coherent
slice); combining authentication and TLS into one milestone here would be a
materially bigger, two-mechanism lift.

## Goals (Milestone 11)

- A single shared cluster secret (one token, one value, no per-node or
  per-client distinction), generated by the operator and distributed as a
  file to every host that runs `keel-controlplane`, `keel-agentd` (in
  control-plane mode), or `keelctl` (in control-plane-routed mode).
- `keel-controlplane` rejects, with `401`, any request on any of its routes
  (`POST /nodes/register`, `POST /nodes/<id>/heartbeat`, `GET /nodes`, the
  named-node forwarding family, the scheduled-apply family) that doesn't
  carry a matching `Authorization: Bearer <token>` header.
- `keel-agentd`'s opt-in TCP listener (the `--advertise-addr` one added in
  Milestone 8) gets the identical check. Its Unix socket is completely
  unaffected â€” see Non-Goals.
- Every outbound network call this project already makes gains the same
  header: `keel-agentd`'s registration/heartbeat calls to the control
  plane, and `keel-controlplane`'s forwarding calls to a node's `keel-agentd`.
- `keel-controlplane`, `keel-agentd`, and `keelctl` each gain a new
  `--auth-token-file <path>` flag. On `keel-controlplane` it's unconditionally
  required. On `keel-agentd` and `keelctl` it's required exactly when the
  existing control-plane flags (`--control-plane-addr` etc.) are present,
  extending the pairing check Milestone 7 already established for
  `--node-id`/`--advertise-addr` â€” plain single-node usage of either binary
  is entirely unaffected.

## Non-Goals (Milestone 11)

- **TLS / wire encryption.** Explicitly deferred to a separate, later
  milestone. This milestone stops an unauthenticated actor from issuing
  commands; it does not stop a sufficiently capable on-path attacker on the
  same LAN segment from reading the token, or any spec body, off the wire.
  Same LAN-trust boundary every prior milestone drew for confidentiality,
  narrowed here only for authorization.
- **Per-node or per-client distinct credentials, or any revocation
  mechanism.** One secret, checked identically everywhere. A leaked token
  compromises the whole cluster equally; there is no way to invalidate one
  node's credential without rotating the shared secret for everyone. This is
  an accepted, explicit limitation, likely superseded later by per-identity
  certificates if and when the deferred mTLS milestone happens, rather than
  built as an intermediate step now.
- **Token rotation or live reload.** The token is read once at process
  startup. Rotating it is entirely an operator procedure: generate a new
  file, copy it to every host, restart every process. No `SIGHUP` handling,
  no polling for file changes.
- **`keel-agentd`'s Unix socket.** Its existing `0600`-permission trust
  boundary, unchanged since Milestone 5, gets no token check at all. This
  milestone only adds checks to the two TCP listeners
  (`keel-controlplane`'s, and `keel-agentd`'s opt-in advertise-addr one).
  Requiring a network credential for local, filesystem-permission-gated
  access would add friction with no corresponding security benefit.
- **Rate-limiting, lockout, or any brute-force mitigation on repeated bad
  tokens.** A network-level control (firewall, VPN, security group) is
  assumed here, consistent with this project never having built anything at
  that layer for any other endpoint.
- **Distinguishing "no header" from "wrong header" in the response**, or in
  timing beyond the token comparison itself (see Error Handling). Both are
  the same generic `401`.

## Architecture

### Wire format: how the token travels

Every hand-built outbound request in the codebase
(`keel-controlplane::http::forward`, `keel-agentd::registration::send_request`,
`keelctl::send_request_tcp`) already constructs its request line and headers
via a `format!` string. Each gains one more header line:

```rust
format!(
    "{method} {path} HTTP/1.1\r\nHost: localhost\r\nAuthorization: Bearer {token}\r\nContent-Length: {}\r\n\r\n{body}",
    body.len()
)
```

No new dependency: `httparse` (already present in both crates) already
parses headers; `ParsedRequest` just needs to retain one of them instead of
discarding everything but `Content-Length`.

### A shared `auth` module (new: `keel-controlplane::auth`, `keel-agentd::auth`)

Small and structurally identical in both crates, matching the project's
established preference for small parallel implementations over cross-crate
sharing (the same choice already made for `keel-controlplane`'s `http.rs`
being a near-exact structural copy of `keel-agentd`'s rather than sharing
code with it):

```rust
pub fn load_token(path: &Path) -> Result<String, String> {
    std::fs::read_to_string(path)
        .map(|s| s.trim().to_string())
        .map_err(|e| format!("failed to read auth token file {}: {e}", path.display()))
}

pub fn check(provided: Option<&str>, expected: &str) -> bool {
    let Some(provided) = provided else { return false };
    let provided = provided.strip_prefix("Bearer ").unwrap_or(provided);
    constant_time_eq(provided.as_bytes(), expected.as_bytes())
}

fn constant_time_eq(a: &[u8], b: &[u8]) -> bool {
    if a.len() != b.len() {
        return false;
    }
    a.iter().zip(b).fold(0u8, |acc, (x, y)| acc | (x ^ y)) == 0
}
```

`constant_time_eq` is hand-rolled rather than pulling in a crate (e.g.
`subtle`) for a four-line function, avoiding a timing oracle on the final
byte-compare without treating "no new dependency" as an absolute rule â€”
contrast with the deferred TLS milestone, where hand-rolling really would be
a mistake.

### `keel-controlplane`: enforcement point

`ParsedRequest` gains one field, captured in the same header-scanning loop
that already extracts `Content-Length`, matched the same
case-insensitive way (`h.name.eq_ignore_ascii_case("authorization")`):

```rust
struct ParsedRequest {
    method: String,
    path: String,
    body: Vec<u8>,
    auth_header: Option<String>,
}
```

`route()` becomes the single choke point, checked before the route match
itself (including before the "no route" fallback, so an unauthenticated
caller learns nothing about which routes exist):

```rust
fn route(request: &ParsedRequest, commands: &Sender<Command>, token: &str) -> (u16, Vec<u8>) {
    if !auth::check(request.auth_header.as_deref(), token) {
        return error_response(401, "missing or invalid auth token".to_string());
    }
    // ... existing match unchanged ...
}
```

`run`/`route` gain `token: Arc<String>`, cloned per connection the same way
`commands` already is. `handle_forward`'s `forward()` and
`handle_scheduled_apply`'s direct forward call also take `token` and attach
the `Authorization` header on the outbound call to a node â€” the control
plane must present the shared secret to whichever `keel-agentd` it relays
to, or that node's own check will reject it.

### `keel-agentd`: enforcement point

Identical shape: `ParsedRequest` gains `auth_header`, but **only the TCP
path checks it**. Two thin wrapper functions keep this unambiguous at the
type level rather than threading a boolean through every call site:

```rust
fn route(request: &ParsedRequest, commands: &Sender<Command>) -> (u16, Vec<u8>) {
    // existing match, completely unchanged -- used by the Unix-socket path
}

fn route_authenticated(request: &ParsedRequest, commands: &Sender<Command>, token: &str) -> (u16, Vec<u8>) {
    if !auth::check(request.auth_header.as_deref(), token) {
        return error_response(401, "missing or invalid auth token".to_string());
    }
    route(request, commands)
}
```

`handle_connection` (Unix socket) calls `route()`, unchanged since
Milestone 5. `handle_connection_tcp` calls `route_authenticated()`. This
makes the Unix-socket exemption a fact about which function is called, not
a runtime branch that could be miswired.

`registration.rs`'s `register_once`/`heartbeat_once` add the
`Authorization` header to their calls to the control plane, using the token
the process loaded at startup.

### `keelctl`: client side

New `--auth-token-file <path>` flag, extracted the same way
`--control-plane-addr`/`--node` already are. The existing pairing check
(today: `--node` without `--control-plane-addr` is a usage error) extends
to also require the token file whenever `--control-plane-addr` is set:

```rust
let token = match &target {
    Target::ControlPlane { .. } => {
        let path = auth_token_file.ok_or("--auth-token-file is required with --control-plane-addr")?;
        Some(std::fs::read_to_string(&path).map_err(|e| format!("failed to read {path}: {e}"))?.trim().to_string())
    }
    Target::Socket(_) => None,
};
```

`send_request_tcp` gains the `Authorization` header when `token` is
`Some`; `send_request` (Unix-socket path) is untouched.

### Startup / flag-parsing changes

- `keel-controlplane/main.rs`: new, unconditionally required
  `--auth-token-file` flag; `panic!` at startup if missing or unreadable,
  matching this binary's existing "fail fast on bad config" style.
- `keel-agentd/main.rs`: `--auth-token-file` joins the existing
  `--node-id`/`--advertise-addr` pairing check â€” the current
  `panic!("--node-id and --advertise-addr are required when --control-plane-addr is set")`
  becomes a three-flag check with the same message shape.
- `keelctl/main.rs`: as above, but a graceful `Err` + `ExitCode::FAILURE`
  (this binary already prefers that over panicking for usage errors, e.g.
  the existing `--node` without `--control-plane-addr` check).
- `401` added to `reason_phrase()` in both `http.rs` files (`"Unauthorized"`).

## Error Handling

- A missing `Authorization` header and a present-but-wrong one produce the
  same response: `401`, body `"missing or invalid auth token"`. The
  response never distinguishes the two, so a caller can't use it to probe
  whether a guessed token was close.
- The auth check happens before any other validation (malformed YAML,
  unknown node id, path/name mismatch) and before the route match itself.
  An unauthenticated request to a nonexistent path still gets `401`, not
  `404` â€” no route information leaks pre-auth.
- `constant_time_eq` guards only the final byte-compare; `check`'s
  `strip_prefix`/`Option`-check steps are variable-time, but they depend
  only on the shape of the caller's input, not on the expected token's
  value, so they leak nothing about the secret.
- A process that fails to load its token file at startup fails loudly and
  immediately (`panic!` for `keel-controlplane`/`keel-agentd`, a graceful
  `Err`/`ExitCode::FAILURE` for `keelctl`) â€” never falling back to running
  unauthenticated. Same "fail fast on bad config" precedent `parse_args`
  already sets for a missing `--node-id`.
- A forwarding hop that fails auth at the *target* node (e.g. a stale token
  file on one node after a rotation that missed it) surfaces as a plain
  relayed `401` through `keel-controlplane`'s existing forward-and-relay
  path â€” no new error type; `handle_forward`/`forward()` already relay the
  target's exact status code verbatim.
- No distinct server-side logging for rejected requests beyond what already
  exists: since no logging framework exists anywhere in this project today,
  a `401` is visible to the caller (and to anyone watching `keel-agentd`'s
  existing `eprintln!`-based registration/heartbeat failure messages) but
  not separately logged on the control-plane side. Worth revisiting only if
  this project ever adds structured logging generally â€” a pre-existing gap
  this milestone doesn't attempt to fix in passing.

## Testing Strategy

- `auth::check` / `auth::load_token`: plain unit tests, no networking â€”
  correct token accepted; wrong token, missing header, empty file, and a
  token file with trailing whitespace (confirming `.trim()`) all handled as
  specified; `constant_time_eq` tested directly for equal, unequal-same-
  length, and unequal-different-length inputs.
- `keel-controlplane::http`: every existing test's `start_test_server`/
  `send_request` helper threads a token through (server configured with a
  known value, `send_request` calls send the matching header) â€” a
  mechanical update to existing tests' plumbing, not new behavior for them.
  New tests: a request with no `Authorization` header, and one with a wrong
  token, both return `401` â€” checked across at least one route from each
  family (`register`, `heartbeat`, `GET /nodes`, a named-node forward, a
  scheduled apply), confirming the single `route()` choke point actually
  covers all of them.
- `keel-controlplane::http::forward`: `start_fake_remote_agentd` gains an
  assertion that the relayed request carries the expected `Authorization`
  header, confirming the control plane attaches its own token when dialing
  a node, not just when receiving one.
- `keel-agentd::http`: the same mechanical update to the TCP-listener tests
  (`start_tcp_test_server`/`send_request_tcp`) plus new missing/wrong-token
  â†’ `401` tests. The existing Unix-socket tests in this same file are
  asserted to need **no changes at all**, proving the socket path is
  genuinely untouched rather than assumed to be.
- `keel-agentd::registration`: existing tests' `start_test_control_plane`
  helper configures a token; assertions confirm registration/heartbeat
  still succeed end-to-end through a now-authenticated control plane. A new
  test drops or corrupts the token on one side and confirms
  `register_once`/`heartbeat_once` surface the resulting `401` as an
  `Err(String)`, and that `spawn`'s existing "any heartbeat failure â†’
  re-register" loop treats it identically to any other failure (no
  special-casing needed â€” confirming the existing catch-all already covers
  this).
- `keel-agentd/main.rs` and `keelctl/main.rs`: flag-parsing unit tests
  mirroring the existing `--node-id`/`--advertise-addr` pairing tests,
  extended to the three/four-flag case, confirming the new
  `--auth-token-file` requirement is enforced the same way.
- VM verification (the real proof, same discipline as every prior
  milestone), across the three-node setup: generate one token file, copy it
  to all three VMs plus the client machine; confirm normal
  apply/get/delete/list all work end-to-end exactly as Milestone 10 left
  them; then confirm a `keelctl` call with a missing or wrong
  `--auth-token-file` is rejected with `401` at the control plane, and a
  `keel-agentd` started with a stale/mismatched token fails to register
  (visible in its `eprintln!` retry loop) against a control plane holding
  the correct one.

## Open Questions / Deferred Decisions

- TLS / wire encryption is next, as its own separately-designed milestone â€”
  this one closes the authorization gap only, not confidentiality.
- Token rotation has no live-reload story; rotating means regenerating the
  file, redistributing it, and restarting every process. Worth a real
  design once a cluster is large enough for a restart-everything rotation
  to be genuinely painful.
- Per-node credential revocation is out of scope entirely for the
  shared-secret model chosen here; if it's ever needed, it likely arrives
  as part of a future mTLS milestone (per-identity certificates are a
  natural revocation point) rather than being bolted onto shared secrets
  first.
- Whether `keelctl` should also accept the token via an environment
  variable (for scripting convenience without a temp file) is left open;
  file-only for now, matching both daemons.
